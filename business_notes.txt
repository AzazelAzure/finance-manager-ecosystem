    # TODO: Create RollingKey model
        # RollingKey.user = charField that holds encrypted user_id from AppProfile
        # RollingKey.current_key = uuid hashed with user_id
        # RollingKey.previous_key = none on creation, then current key each time the current key is used.
        # RollingKey.refresh = Bool, default False.  Set this to true if a refresh token is used to log in.
            # What this will do is whenever the current key is used, a new key will be generated.
            # When the new key is generated, it will set the old current key as the previous key.
            # The idea is on login, the user is passed the current key.  
            # The current key will be required for any future requests.
            # Once a request is made, the current key will be set to the previous key, and a new key generated.
            # Designed this way to ensure packetloss doesn't lock user out.
                # If the previous key is used, it will only ever return the current key.
                # Since any time the current key is used, and the the previous key is overwritten
                # Ensures even if the previous key is compromised, it cannot be used to decrypt user data.
                # Also ensures that if the user uses the current key, that compromised key will no longer function.
        
        # Proposed Flow:
            # User signs in with token or password
            # RollingKey.curent_key is generated and returned to user
            # For all future requests, the current key will be required.
                # When a request is made, the current passed is hashed and checked again the rolling keys for user.
                # If incorrect, returns 403 Forbidden, and no further changes.
                    # Caveat, if key passed is previous_key, then return current_key.
                # If correct, request is processed, and a new current key is generated and passed to user. 
                    # The old current key is then set to previous_key

        # General thoughts:
            # Since app profile will always be accessible, this could cause some issues, but I cannot solve everything
            # If the user is verified by login token, that's on them.  
            # However, this would require both the bearer token and a correct current_key to actually do anything
            # This would, in theory, make any sort of long term access moot.  
            # However, should make a session table as well.
