## Step 1 — Load context

Read `strategy/automations/context/daily_context.md` first.

This file is generated by `scripts/dev/gather_doc_context.sh`, which runs as a local cron job before this automation. It contains:
- Git log for all repos (last 24h)
- Merged PR lists for parent, API, and web repos (last 30 each)
- Current plan registry Draft/Planning section
- Runtime Signup Sheet snapshot (live VPS state)
- Stale pattern scan results (Reflex refs, old branch names, deprecated tools)

- `git log --since="24 hours ago" --oneline` (parent + submodules)
- `gh pr list --state merged --limit 30`
- `governance/plan_registry.md`
- `design_docs/30_Releases/Runtime_Signup_Sheet.md`

## Step 2 — Plan registry reconciliation

Open `governance/plan_registry.md`.

For each plan in the **Draft / Planning** section:

1. Find the plan's README at its `plan_root` path (in the registry row notes field)
2. Read the plan README — identify what PRs it references
3. Cross-check against the merged PR lists in the context file
4. Apply the following decision logic:
   - **All referenced PRs merged → move to "Recently Completed"** with today's date and PR URLs
   - **Some PRs merged → update the registry row notes** field to reflect partial completion
   - **No PRs merged, plan predates 30 days → flag in sweep report** for HitM review
   - **No PRs merged, plan is recent → leave in place**, no change

When moving a plan to Recently Completed, add a row using this format:
```
| `PLAN_ID` | S1.B | YYYY-MM-DD | [one-line strategic impact] | [PR URL(s)] |
```

Update the `governance/plan_registry.md` **Last updated:** header line to today's date.

Also check the **In Progress** section: if a plan's notes say "pending close-out" or similar and all its PRs have merged, move it to Recently Completed as well.

## Step 7 — Anomaly queue triage

Read the **Anomaly Queue** section from `strategy/automations/context/daily_context.md`.

For each unreviewed anomaly log:

1. **Quick resolution check:** Look at the git log (from the context file) for recent commits touching the file/path mentioned in the anomaly. If a commit in the last 7 days clearly addresses the described issue, mark the anomaly `resolved` with a one-line note citing the commit SHA.

2. **Duplicate check:** If two anomaly logs describe the same underlying issue in the same file, note the duplicate in the sweep report. Don't merge them — flag for HitM to close the older one.

3. **Owner assignment:** For each anomaly still `unreviewed` after the resolution check, suggest a dispatch target:
   - `Cursor` — code-level fix, belongs in a sprint task or slice
   - `Claude Code` — governance or planning doc concern
   - `HitM` — judgment call, external service, or resource decision
   - `VPS manual` — requires SSH investigation or container-level change
   - `Defer` — real but low-stakes; add to next cleanup pass

**Write permissions for this step:**
- `strategy/anomalies/*.md` — update `status:` field from `unreviewed` to `resolved` only when there is clear commit evidence. Do not mark `wont-fix` — that's HitM's call.
- Do not delete anomaly files.

**Include in sweep report (Step 6):**

```markdown
## Anomaly Queue

| File | Severity | Plan context | Status after triage | Dispatch to |
|---|---|---|---|---|
| YYYY-MM-DD_slug.md | medium | PLAN_ID / T## | unreviewed → Cursor | Add auth guard |
| YYYY-MM-DD_slug.md | low | PLAN_ID / T## | resolved (abc1234) | — |
```

If there are no unreviewed anomalies: write "Anomaly queue: clear."

## Constraints

- **Targeted edits only.** One wrong reference corrected is better than one correct paragraph rewritten.
- **Don't close plans prematurely.** If PR evidence is ambiguous — plan README lists a PR but context file doesn't show it merged — update notes, don't close.
- **Don't infer VPS state from git history.** VPS state comes from `design_docs/30_Releases/Runtime_Signup_Sheet.md` only.
- **Don't touch strategic decisions.** Plan statuses are facts about what shipped. Phase transitions and gate outcomes belong to HitM.
- **Write the sweep report even if nothing changed.** A clean sweep report confirms the automation ran.
- **Do not commit.** HitM reviews the sweep report and commits, or the daily summary handles the commit.
